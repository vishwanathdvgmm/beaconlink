# 132 - v0.1.0 BeaconLink Protocol (BLP)

- **Status:** Planned
- **Version:** v0.1.0
- **Milestone:** BeaconLink Protocol (BLP)
- **Estimated Duration:** 3–5 Weeks
- **Last Updated:** 2026-07-20

---

# 1. Overview

The objective of **v0.1.0** is to implement the **BeaconLink Protocol (BLP)**.

The protocol is the communication foundation of the entire platform.

Every component—including the Agent, Relay, CLI, API Gateway, and future SDKs—communicates using BLP.

This release delivers a complete protocol implementation but does **not** include production networking services.

---

# 2. Goals

This milestone establishes:

- Protocol specification implementation
- Binary packet encoding
- Binary packet decoding
- Message definitions
- Session state machine
- Version negotiation
- Handshake implementation
- Error handling
- Protocol validation
- Unit testing

---

# 3. Out of Scope

The following are intentionally excluded.

- Relay server
- Agent
- Runtime abstraction
- Deployment engine
- API Server
- Console
- Database
- Authentication
- Authorization
- Certificate management

Those components consume the protocol but are implemented later.

The protocol implementation should provide:

- Packet encoder
- Packet decoder

# 8. Packet Format

Implement:

- Header
- Version
- Message Type
- Flags
- Session ID
- Payload Length
- Payload
- Checksum (if defined)

Packet structures must exactly match the protocol specification.

---

# 9. Message Types

Initial messages include:

- Handshake
- Handshake Response
- Ping
- Pong
- Error
- Heartbeat
- Register
- Acknowledge

Additional message types may be introduced later without breaking compatibility.

---

# 10. Version Negotiation

The protocol must support:

- Client version
- Server version
- Negotiation
- Unsupported version detection

Connections with incompatible protocol versions must be rejected cleanly.

---

# 11. Session Lifecycle

Implement the protocol session state machine.

```text
Disconnected

↓

Connecting

↓

Negotiating

↓

Authenticated (future)

↓

Ready

↓

Closing

↓

Closed
```

Authentication remains a future milestone.

---

# 12. Handshake

The handshake should establish:

- Protocol version
- Session identifier
- Feature negotiation
- Connection capabilities

The handshake should not perform authentication.

---

# 13. Serialization

Protocol serialization must be:

- Deterministic
- Efficient
- Backward compatible
- Strictly validated

Malformed packets must never panic the application.

---

# 14. Error Handling

Implement protocol errors.

Examples:

- Invalid Version
- Invalid Packet
- Unknown Message
- Invalid State
- Invalid Payload
- Session Closed
- Decode Failure
- Encode Failure

Errors should map directly to the protocol specification.

---

# 15. Validation

Validation should include:

- Header validation
- Length validation
- Payload validation
- State validation
- Version validation

Invalid input must return descriptive errors.

---

# 16. Public API

The public protocol package should expose only stable APIs.

Example:

```go
packet, err := protocol.Encode(message)

msg, err := protocol.Decode(packet)
```

Implementation details remain internal.

---

# 17. Performance Goals

Target characteristics:

- Zero unnecessary allocations
- Streaming-friendly
- Minimal copying
- Efficient parsing

Performance optimization must not compromise readability.

# 24. Risks

Potential risks include:

- Protocol compatibility issues
- Inefficient packet parsing
- Future extensibility constraints
- Overly complex message definitions

These should be addressed before moving to networking components.

---

# 25. Success Criteria

The protocol should be capable of supporting two independent applications exchanging BeaconLink messages without requiring any Relay or Agent implementation.

This milestone is successful when BLP is stable, fully tested, documented, and ready to serve as the communication layer for all subsequent platform components.

---

# 26. Summary

Version **v0.1.0** delivers the first operational component of BeaconLink: the **BeaconLink Protocol (BLP)**.

While it introduces no end-user functionality, it establishes the standardized communication layer that all future services—including the Relay, Agent, API Server, and SDKs—will build upon. A robust, well-tested protocol at this stage minimizes future compatibility issues and provides a stable foundation for the remainder of the implementation roadmap.
